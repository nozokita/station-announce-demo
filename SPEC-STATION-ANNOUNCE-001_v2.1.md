# 多言語駅放送WEBアプリケーション 仕様書

**文書番号:** SPEC-STATION-ANNOUNCE-001  
**作成日:** 2026年3月27日  
**バージョン:** 2.1

---

## 1. プロジェクト概要

### 1.1 目的

鉄道の各駅において、日本語・英語・韓国語・中国語の4言語でアナウンス放送を行うためのWEBアプリケーションを開発する。放送文例の登録、翻訳、音声合成、定期放送のスケジュール管理までをブラウザ上で完結させる。

### 1.2 対象ユーザー

本システムは社内専用であり、本部の管理者および各駅のオペレーターが使用する。外部への公開は行わない。

### 1.3 運用規模

初期は数駅でスモールスタートし、将来的に最大50駅程度までの拡張を想定する。

---

## 2. システム構成

### 2.1 全体像

本システムは「サーバー1台＋端末のブラウザ」というシンプルな構成を採用する。複雑なマイクロサービスや複数のAWSマネージドサービスへの依存を避け、運用・保守のしやすさを最優先とする。

### 2.2 各構成要素の役割

- AWS Lightsail（サーバー）
- Python FastAPI + APScheduler（アプリケーション + スケジューラ）
- PostgreSQL（データベース）
- Nginx（リバースプロキシ）
- Amazon Polly（音声合成）
- Amazon Translate（自動翻訳）
- Amazon S3（音声ファイル保管）
- 端末ブラウザ（放送操作、再生、キャッシュ）

### 2.3 データの流れ

- 放送文例登録時: 入力 → 翻訳 → 修正 → 音声生成 → 保管
- 即時放送時: 文例選択 → 再生言語選択 → 音声再生 → ログ送信
- スケジュール放送時: サーバー時刻到来 → 端末へ再生指示 → 再生結果記録

## 3. 機能一覧

### 3.1 ユーザー認証

ログインID・パスワードで認証を行う。JWTによりAPIおよびWebSocket接続を認可する。  
ロールは以下の2種類:

- **admin（管理者）**: 全機能にアクセス可能
- **operator（オペレーター）**: 所属駅中心の運用機能にアクセス可能

### 3.2 駅管理（v2.1更新）

管理者が駅の追加・編集・無効化を行う。  
**駅コード（code）は必須要件から削除**し、駅名中心で運用する。  
例: 名古屋駅、岐阜駅。

### 3.3 放送文例管理（v2.1更新）

管理者が放送文例を登録・編集・削除する。  
文例には「全駅共通」または「特定駅専用」を指定できる。

- 削除は**論理削除**とする
- 論理削除文例は文例一覧に表示しない
- ログ画面では対象文例を「（削除済み）」表示とする
- 関連お気に入りは削除時に除外する
- 関連スケジュールは削除文例に紐づくため、実行不可として無効化または削除扱いにする
- 復元は原則不可（再作成運用）

### 3.4 お気に入り管理（v2.1更新）

各駅のオペレーターが文例を星トグルでお気に入り登録/解除できる。  
お気に入りは駅単位で管理し、表示順（sort order）を保持する。

- 追加
- 解除
- 並び順管理（優先表示）

### 3.5 放送実行（即時放送）（v2.1更新）

オペレーターが文例一覧から放送を選択し、再生言語を選んで放送を実行する。  
全言語順番再生は以下を必須とする:

- 順序: **JA → EN → KO → ZH**
- 言語間: **1秒無音**

モバイル要件（特に iPhone/Safari）:

- 全言語順再生が1言語目で停止しないこと
- 連打/再操作時に旧キューが残留して後から一括再生されないこと
- 再生失敗時にユーザーへ明確なエラー通知を出すこと

※ 再生方式（Web Audio API / HTMLAudioElement等）は実装依存とし、動作要件を満たすことを優先する。

### 3.6 スケジュール放送（v2.1更新）

放送文例に対して定期放送スケジュールを設定できる。  
スケジュールタイプは `once` / `weekly` / `interval` を基本とするが、UI要件として以下を定義する:

- `weekly` を必須機能として維持するか再検討対象とする
- 現場運用と乖離する場合、`once` / `interval` 中心に要件整理する
- 採用方針確定時に、入力項目・バリデーション・API定義を統一更新する

### 3.7 サーバー障害時の端末表示

WebSocket接続が切断され再接続失敗が継続した場合、画面上部にアラートバナーを表示する。  
文言例:

「サーバーに接続できません。スケジュール放送は停止中です。即時放送はローカルキャッシュの範囲で利用可能です。」

### 3.8 放送ログ（v2.1更新）

いつ、どの駅で、どの文例が放送されたかを記録する。  
ステータスは `success` / `missed`。

- 管理者は全駅ログを閲覧可能
- オペレーターは自駅ログのみ閲覧可能
- 日付フィルタで絞り込み可能
- 削除済み文例はログ上「（削除済み）」表示

**保持期間: 1ヶ月**（v2.0の3ヶ月から変更）

時刻要件:

- 表示時刻はJST（Asia/Tokyo）基準
- 日付フィルタ判定もJST日付基準
- UTC保存時でも表示・検索・集計のユーザー挙動はJST一致とする

## 4. 画面設計

### 4.1 画面一覧（全7画面）

1. ログイン画面
2. 放送文例一覧（メイン画面）
3. 放送文例 登録・編集画面（admin）
4. スケジュール管理画面
5. 放送ログ画面
6. ユーザー管理画面（admin）
7. 駅管理画面（admin）

### 4.2 各画面の概要

- ログイン画面: 認証、音声有効化操作
- 放送文例一覧: 駅選択、星トグル、放送実行、接続状態表示
- 文例登録・編集: 登録/編集/削除、翻訳、音声生成、試聴
- スケジュール管理: 一覧、追加、切替、削除、タイプ別入力
- 放送ログ: JST時刻表示、JST日付フィルタ
- ユーザー管理: 追加/編集/削除
- 駅管理: 追加/編集/無効化（駅コード必須なし）

## 5. データベース設計

PostgreSQLを使用し、主要テーブルで構成する。  
TIMESTAMP系はTIMESTAMPTZを使用し、表示・検索時はJSTを基準とする。

### 5.1 テーブル定義（要点）

#### stations

- id
- name
- is_active
- created_at

※ v2.1で `code` 必須要件は削除。

#### users

- id
- username
- password_hash
- role（admin/operator）
- station_id
- created_at

#### broadcasts

- id
- station_id（NULLで全駅共通）
- title
- text_ja / text_en / text_ko / text_zh
- audio_ja / audio_en / audio_ko / audio_zh
- is_deleted（論理削除）
- created_by / created_at / updated_at

#### station_favorites

- id
- station_id
- broadcast_id
- sort_order

#### schedules

- id
- station_id
- broadcast_id
- languages
- schedule_type（once/weekly/interval）
- once_at / days_of_week / time_of_day / interval_min
- start_date / end_date
- is_active
- created_by / created_at

#### broadcast_logs

- id
- station_id
- broadcast_id
- schedule_id
- played_at
- status（success/missed）
- device_info

### 5.2 削除時データ整合ルール（v2.1更新）

- 放送文例の論理削除時:
  - 文例一覧から非表示
  - ログは参照可能（文例名は「削除済み」）
  - お気に入りから除外
  - 関連スケジュールは実行不可化

### 5.3 スケジュールタイプ別要件（v2.1更新）

- `once`: 指定日時で1回
- `weekly`: 曜日・時刻指定（継続採用は再検討対象）
- `interval`: 間隔指定（現場運用重視）

採用方針確定時に、UI/バリデーション/APIの整合を必須とする。

## 8. コスト見積もり

### 8.1 初期（数駅運用）

- Lightsail: 約 $12 / 月
- S3: 少額
- Polly: 少額
- Translate: 少額

### 8.2 拡大時（50駅運用）

- Lightsail: 約 $24 / 月
- S3/Polly/Translate: 利用量に応じて増加

- Pollyはキャッシュ戦略により登録/修正時中心の課金
